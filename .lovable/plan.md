## Діагноз

Сам застосунок **не зламаний**:

- Dev-сервер у пісочниці відповідає `200`, `vite ready`.
- `bunx tsc --noEmit` — чисто.
- `bun run build` — успішний (`✓ built in 3.60s`, `dist/server/index.mjs` та всі чанки згенеровано).
- Runtime-помилок немає.

Помилка, яку ви бачите в Preview, — інфраструктурна. У логах preview-воркера повторюється:

```
[proxy-worker2] Error: Worker bundle not found:
  dwl:pre:9e9a2736-...:9132fdb7:_worker_bundle.json
GET /me → 404
```

Тобто proxy шукає preview-бандл для коміту `9132fdb7`, але цього бандла немає в сховищі (він не доїхав / був витерт / попередня збірка зірвалась). Звідси повідомлення «Preview has not been built yet». Жодних правок коду це не лікує — потрібно змусити Lovable перезібрати preview-бандл для поточного коміту.

## План

1. **Зробити мінімальну незначущу зміну**, щоб тригернути новий preview-build з новим SHA (наприклад, оновити коментар-маркер у `.env` чи додати порожній рядок у `ITERATION-1-PUSH-DEPLOY.md`). Жодної логіки/UI не чіпаємо.
2. Після збірки перевірити `stack_modern--server-function-logs` (deployment=`preview`) — там має зникнути `Worker bundle not found` і з'явитись `GET /me → 200`.
3. Якщо після ретригера лог знову покаже ту саму помилку «bundle not found» для нового SHA — це вже на стороні Lovable infra (proxy / storage), і тоді потрібно звернутись у підтримку Lovable; код-фікса з нашого боку не існує.

## Що НЕ робимо

- Не правимо `src/server.ts`, `vite.config.ts`, роутер, RLS чи будь-яку бізнес-логіку — збірка та локальний рантайм здорові, це не SSR-500-кейс із гайду.
- Не публікуємо (`preview_ui--publish`) — це окремий продакшен-деплой, не виправить preview-бандл.
- Не перезапускаємо dev-server (він і так живий).

Підтвердіть — і я застосую крок 1 та перевірю логи.
